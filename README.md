# CPlayer 在线版（保真后台 + 歌单）

基于 [ChKSz/CPlayer](https://github.com/ChKSz/CPlayer) 修改：

- 原生 `<audio>` 路径，利于手机浏览器后台播放
- 不压缩/不重编码，不削弱音质
- 当前播放列表、自建歌单和播放模式会保存在浏览器本地
- 支持顺序、单曲循环、列表循环和随机播放
- 支持最近播放、歌单排序、备份导入/导出和播放失败自动跳过
- 支持恢复上次歌曲与播放进度，以及可跨刷新的睡眠定时

## 日常使用

1. 在“搜索添加”中搜索歌曲。点击整行会加入并立即播放。
2. 点“加入”只放进当前播放列表，不会打断正在播放的歌曲。
3. 点“歌单”会把歌曲收藏到自己创建的歌单。
4. 在“音乐资料库”中管理歌单、最近播放和备份。
5. 在“设置”中选择睡眠定时；到点后播放器会自动暂停。

当前播放列表和自建歌单是两套数据：前者是这次准备播放的队列，后者是长期收藏。清空当前播放列表不会删除自建歌单。

播放时会优先请求 JyMaster 音质，但每首歌实际可用的音源由上游服务决定。质量徽标会优先显示上游 API 标注；没有可靠标注时只按 URL/码率保守推断，并显示“音质未标注”而不冒充母带。

## 数据与离线

- 未配置云端时，数据保存在当前浏览器的 IndexedDB/localStorage 中；账号登录是可选的。
- 播放进度只保存在当前浏览器。刷新或重新打开后，点击播放即可从上次位置继续；播放器不会绕过浏览器限制自动发声。
- 建议定期在音乐资料库中导出歌单备份；清理浏览器网站数据前先备份。
- 已打开过的播放器界面可以离线启动；搜索、歌词、获取歌曲地址和在线播放需要网络，也依赖第三方 `api.chksz.top` 服务。该服务临时不可用时，在线功能会失败，但已保存在浏览器本地的歌单和播放队列仍可查看与管理。
- “播放器设置”中可以填写个人 ChKSz API 密钥和自定义 API 地址。默认仍使用 `https://api.chksz.top/api`，不填写密钥时保持原有行为。密钥只保存在当前浏览器的 `localStorage` 中，请只在自己的设备上使用；请求音乐服务时，密钥会作为 `apikey` 查询参数发送给所填地址。
- “播放器设置”中的“账号与云同步”可以用邮箱密码注册/登录。登录后只同步自建歌单和歌曲顺序；当前播放列表、最近播放、播放进度、睡眠定时、API 地址和 API 密钥不会上传。云端不可用或断网时，本机歌单仍可使用，待同步修改会在恢复网络后重试。当前仓库的 `js/cloud-config.js` 仍是空配置，真正启用云端前需要创建 Supabase Free 项目并只填写公开的项目 URL 与 publishable/anon key，绝不能填写 service-role key。
- Supabase 账号功能只接受 HTTPS 项目地址。Free 计划本身可免费使用，但有额度和项目暂停规则；本地代码不会自动创建项目、绑定银行卡或升级付费计划。创建真实项目、填写公开配置和上线前需要单独确认。
- 已安装的播放器检测到新版本并完成缓存切换后，会显示“播放器已更新”。点“刷新”会先保存当前队列和可恢复的播放进度，再加载新版本；点“稍后刷新”不会打断当前使用。
- `playlist.js` 是可选的本地歌单入口。默认文件为空，需要时可用导出的同名文件替换。

页面左下角显示当前构建版本；Service Worker 的缓存修订号和歌单备份版本只用于内部兼容，不等同于产品版本。若更新后仍显示旧构建，请刷新页面或完全关闭后重新打开 PWA。

## 本地验证

验证环境需要 Node.js 22 或更高版本，以及 Python 3。项目的本地脚本和 GitHub Actions 使用同一 Node.js 主版本；Node.js 18 会在质量门禁启动前失败，因此不要继续使用旧版本。

第一次验证时，在项目目录运行：

```powershell
npm ci
npx playwright install chromium
```

以后每次准备提交或发布，只需要运行：

```powershell
npm run verify
```

它会依次检查 CSS 和 Supabase 浏览器依赖构建、单元测试、JavaScript 语法、关键功能契约、
依赖安全、GitHub Pages 产物、桌面与移动浏览器流程，以及 Git 文本格式。
浏览器回归直接从 `output/pages` 运行，这就是工作流随后上传的目录，不再
另做一份文件清单。任意一步失败都会停止并指出失败环节。浏览器失败时的
截图和跟踪记录位于 `output/playwright/`；两个 `output/` 子目录都不会提交。

## 发布与回退

只想查看最终静态站点包含哪些文件时，在项目目录运行：

```powershell
npm run build:pages
```

命令会重新生成 `output/pages`，并输出文件数和总字节数。推送 `main` 会
影响线上 GitHub Pages，必须先得到明确确认；远端 quality job 会重新运行
`npm run verify`，并把通过浏览器回归的同一目录交给 deploy job。

当前浏览器数据库已经是 DB v5。不能直接把线上代码退回只会打开 DB v4
的旧提交，否则已有队列、歌单和云同步待办记录会暂时无法读取。准备回退前先检查目标提交：

```powershell
npm run check:rollback -- <目标提交号>
```

只有命令通过，且该目标再次通过 `npm run verify`，才可以制作新的向前回退
提交。不要清空用户网站数据，也不要用强制推送改写远端历史。

仅供个人学习研究，请合法合规使用。
