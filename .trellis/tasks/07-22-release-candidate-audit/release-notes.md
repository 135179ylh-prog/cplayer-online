# CPlayer 5 本地发布候选说明

## 本轮结果

这个候选版本把播放器从“功能可用”整理为有完整本地质量门禁、故障恢复、移动适配和可回退更新流程的版本。当前只创建本地提交，没有推送或部署。

## 主要变化

- 搜索失败可保留输入和结果面板并重试；第三方服务异常、离线、API 密钥无效或额度用完都有明确提示。
- 用户可以在设置中填写自己的 ChKSz API 地址和密钥。密钥只保存在当前浏览器，但请求时会作为 `apikey` 发送给所选 API。
- 队列、自建歌单、最近播放、备份导入和播放进度具备浏览器存储回归；非法备份不会覆盖已有数据。
- PWA 更新会提示用户选择刷新，刷新前保存队列和播放进度；旧缓存升级后仍可离线打开应用壳。
- 桌面、手机、355px 窄屏和 440px 宽折叠屏拥有一致的弹窗、标签、面板、键盘和触控行为。
- 主应用脚本已从超大 HTML 中拆成独立 `js/app.js`，可以直接做语法检查和独立缓存。
- PWA 描述不再保证“超清母带”；实际音质只按上游可靠标注或保守推断显示。

## 数据兼容

- IndexedDB 和 localStorage 的结构没有迁移，已有队列、歌单、最近播放、API 设置和可恢复播放进度继续使用。
- 不需要也不建议为了升级清空网站数据。
- 清理浏览器数据前，建议先在音乐资料库导出歌单备份。

## 本地发布前检查

在项目目录执行：

```powershell
npm ci
npx playwright install chromium
$env:PW_PORT='4175'
npm run verify
```

预期：八层门禁全部通过，依赖审计 0 个高危漏洞，浏览器回归无失败。这里使用 4175，因为本机 4173 和 4174 已被其他预览服务占用；以后可换成任意空闲端口。

## 发布步骤

以下操作会影响远端，必须得到用户明确确认后执行：

1. 确认工作区干净，记录准备发布的提交号。
2. 推送 `main`；GitHub Pages workflow 会先执行 quality job，失败时 deploy 不会启动。
3. 等待 Pages 部署完成，打开线上地址确认构建标记、主界面、设置和搜索入口。
4. 在实体华为设备按 `device-validation.md` 完成后台播放、锁屏、折叠布局和更新恢复。

## 已知限制

- 在线搜索、歌词和音频地址依赖第三方 ChKSz 服务，不把其实时可用率作为本地 CI 成败条件。
- 浏览器不允许页面在没有用户操作时自动发声；恢复进度后仍需要用户点击播放。
- Chromium 模拟不能证明真实设备的省电策略、锁屏媒体控制或 TalkBack 播报效果。
- 当前自动门禁以 Chromium 为主，未把 WebKit/Firefox 纳入每次发布。

## 回退

1. 选择此前完整门禁通过的提交，回退发布候选提交；不要清空用户浏览器数据。
2. 在回退代码上运行 `npm run verify`。
3. 经用户确认后推送回退提交。对应 `sw.js` 会安装自己的缓存版本，并清理其他 `cplayer5-*` 缓存。
4. 线上确认应用壳、设置、队列、歌单与更新提示；若第三方 API 正在故障，只验证本地功能与明确错误反馈。

回退不会迁移或删除 IndexedDB/localStorage。只有用户主动清理站点数据才会丢失本地歌单和设置。
