# PRD - 发布候选审计与交付

## 目标

对长期 Goal 的五个阶段做逐项证据审计，整理一个可复测、可发布、可回退的本地发布候选。自动化能证明的内容必须有代码或命令证据；真实手机后台播放、锁屏控制和读屏等本地无法替代的项目，必须形成用户可执行的设备清单，不能伪装成已验证。

## 已确认事实

- 当前分支已完成质量基线、核心链路可靠性、API 密钥、搜索恢复、PWA 更新、响应式/无障碍和主模块拆分任务。
- 最新完整门禁为 8/8 层通过；82 个浏览器用例中 80 通过、2 个按设计跳过，覆盖桌面、常见手机、355px 窄屏和 440px 宽折叠屏。
- 队列、备份、歌单、最近播放、播放失败、API 鉴权、旧缓存升级和离线恢复均已有确定性浏览器证据。
- GitHub Pages 工作流在部署前运行同一 `npm run verify`，并使用明确的静态资源白名单。
- 当前不能替代华为 Pura X 的实体设备锁屏/后台播放、系统媒体通知和 TalkBack 读屏验证。
- `manifest.json` 仍把产品描述为“超清母带在线音乐播放器”，与“不在上游未标注时保证母带”的现有质量契约冲突。

## 需求

### R1 Goal 完成度矩阵

- 对长期 Goal 五项完成标准逐项列出当前权威证据、结论和剩余限制。
- 对搜索、播放、队列、歌单、最近播放、备份、离线和更新八条核心链路列出自动化测试所有者。
- 不使用旧任务中的过期测试计数代替当前完整门禁结果。

### R2 发布候选元数据与说明

- PWA manifest 使用不夸大音质能力的真实描述。
- 生产预缓存资源变更后提升 Service Worker 缓存版本，并同步静态契约。
- 提供面向用户的发布说明：主要变化、数据兼容、API 密钥、PWA 更新方式、已知限制。

### R3 发布与回退

- 给出从本地检查到 GitHub Pages 发布的最小步骤，并明确哪些动作需要用户确认。
- 回退说明必须包含代码回退、Service Worker 缓存切换、数据影响和验证步骤。
- 不在本任务 push、部署、打 tag 或触发远端 workflow。

### R4 实体设备验证

- 提供华为 Pura X/Android PWA 的后台播放、锁屏、系统媒体控制、网络切换、更新恢复和宽折叠布局清单。
- 提供 TalkBack/键盘验证步骤和明确的通过标准。
- 清单必须区分“发布前必须”与“建议补充”，并允许记录设备、浏览器和系统版本。

### R5 最终本地门禁

- 重新执行完整 `npm run verify`，记录当前计数、视口、缓存版本和依赖审计。
- 工作区在本地提交、任务归档和 journal 后保持干净。

## 验收标准

- [x] Goal 五项标准和八条核心链路都有当前证据或明确未验证结论。
- [x] manifest 不再宣称“超清母带”，Service Worker 与静态契约同步。
- [x] 发布说明包含功能、兼容、已知限制和用户可理解的更新说明。
- [x] 回退方案说明代码、缓存、数据和验证，不依赖破坏性数据操作。
- [x] 实体华为设备与 TalkBack 清单可直接执行，未完成项不写成通过。
- [x] `npm run verify` 八层全部通过，关键四视口保持通过。
- [x] 创建本地提交、归档和 journal；不推送、不部署。

## 范围外

- 不在没有用户确认时 push、部署、创建 tag 或 GitHub Release。
- 不把 Chromium 模拟当作真实 Android 电源管理或锁屏媒体控制证据。
- 不引入 WebKit/Firefox 全套回归；本项目当前发布门禁明确以 Chromium 与华为 Android 使用场景为主。
- 不修改业务数据 schema、API、播放策略或界面布局。
