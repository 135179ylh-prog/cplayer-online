# PRD - 主应用模块拆分与性能基线

## 目标

把 `index.html` 中约 28 万字符的主应用 ES module 原样提取到 `js/app.js`，建立独立的脚本检查和 PWA 缓存边界。播放器的界面、功能、数据格式、API 行为和启动时序保持不变，为后续维护与性能优化降低单文件改动风险。

## 已确认事实

- 当前 `index.html` 为 388269 字节，主模块约 281491 字符，页面总计约 7900 行。
- 主模块只有一个静态 import：`./js/core-utils.js`。外置到 `js/` 后必须改成相对新模块位置的 `./core-utils.js`。
- `type="module"` 默认延迟到 HTML 解析完成后执行，因此脚本标签虽然位于后续弹窗标记之前，外置后仍能读取完整 DOM。
- `playlist.js` 与 `color-thief.umd.js` 是先加载的经典脚本，主模块仍依赖它们提供的全局能力；不能改变加载顺序。
- Service Worker 当前预缓存 `index.html` 和 `js/core-utils.js`，但还没有 `js/app.js`。
- Pages 工作流已经复制整个 `js/` 目录，新增生产脚本会进入发布产物。
- 80 个 Playwright 用例已经覆盖启动、API、播放、队列、歌单、备份、更新、响应式与无障碍，可作为等价迁移保护。

## 需求

### R1 模块边界

- `index.html` 只保留外部 `<script type="module" src="./js/app.js">` 入口，不再包含主应用内联模块。
- `js/app.js` 保留现有主模块的全部逻辑和初始化顺序，只调整因文件位置变化而必须修改的 import 路径。
- 不引入打包器、框架、TypeScript、动态 import 或新的运行时依赖。

### R2 构建、检查与静态契约

- `check:module` 直接对生产文件 `js/app.js` 执行 `node --check`，不再从 HTML 用正则提取临时文件。
- 静态功能验证区分 DOM 标记与应用脚本来源；业务契约从 `js/app.js` 检查，HTML 结构契约继续从 `index.html` 检查。
- 门禁必须阻止主模块被重新内联、入口缺失、import 路径错误或生产脚本未进入缓存。

### R3 PWA 缓存与升级

- `js/app.js` 加入 Service Worker 核心资源，缓存版本提升到新的唯一修订号。
- 老版本到新版本升级测试明确证明当前缓存包含 `/js/app.js`。
- 在线首启、离线壳重载和更新后离线重载均能完成应用初始化。

### R4 性能基线

- 记录迁移前后 `index.html`、`js/app.js` 和合计字节数。
- 验证浏览器只请求一个主应用模块，且资源类型为 JavaScript。
- 不把本机毫秒耗时设为硬门禁，也不宣称拆分必然降低首次加载总字节数；本阶段只建立可复测的资源边界与基线。

## 验收标准

- [x] `index.html` 不含主应用内联模块，`js/app.js` 是唯一主应用模块入口。
- [x] 外置前后业务逻辑除 import 路径外保持等价，主模块可被 Node 直接解析。
- [x] Service Worker 预缓存 `js/app.js` 并使用新缓存版本，升级测试检查该资源存在。
- [x] 静态契约从正确文件读取，并能防止回退为内联模块或漏发脚本。
- [x] 迁移前后字节数和浏览器资源边界写入验证记录。
- [x] `npm run verify` 八层门禁全部通过；完整浏览器套件无新增跳过或失败。
- [x] Trellis 前端规范记录主模块、语法检查和缓存同步契约。
- [x] 只创建本地提交；不推送、不部署。

## 范围外

- 不拆分 `js/app.js` 内部的业务模块，不重命名现有函数或状态。
- 不改 UI、文案、存储 schema、API 请求、播放策略、响应式布局或无障碍行为。
- 不压缩或混淆 JavaScript，不引入 Vite/Webpack/Rollup。
- 不把网络与机器波动较大的启动耗时作为验收阈值。
