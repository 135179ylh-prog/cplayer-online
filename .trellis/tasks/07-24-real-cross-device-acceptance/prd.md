# 真实跨设备验收

## Goal

在真实桌面和手机浏览器、同一真实账号上验证自建歌单的新建、编辑、排序、删除、离线恢复和并发冲突，保留可核验证据并清理测试数据。

## Requirements

- 使用正式 GitHub Pages、现有 Supabase Free 项目和用户真实账号，不用测试
  Supabase、伪造真实用户或 service-role/admin 密钥。
- 设备 A 为用户日常桌面 Chrome，设备 B 为真实手机浏览器（优先日常使用的
  华为 Pura X）；两者必须是独立浏览器存储，不能用同一 Playwright context
  冒充两台设备。
- 密码只由用户在手机上输入。验收记录不得包含密码、token、完整会话数据、
  API 密钥或 API 地址。
- 使用唯一测试歌单 `跨设备验收-20260724-*` 和现有播放队列中的普通歌曲；
  不修改用户已有歌单。完成后从设备删除该测试歌单并记录 tombstone 事实。
- 在线主路径必须双向验证：A 创建/添加/排序后 B 收到；B 修改后 A 收到；
  名称、歌曲 id、数量和顺序一致，状态中心均显示待同步 0、冲突 0。
- 本地优先路径必须真实验证：B 断网修改后本机歌单立即可用、outbox 状态
  显示待同步；恢复网络后自动发送并在 A 拉取。
- 冲突路径必须真实验证：B 离线基于旧版本修改，A 在线提交另一修改；B
  恢复网络后必须显示冲突且保留双方版本，未选择前不得静默覆盖。
- 用户明确选择“使用本机”或“使用云端”后，两端最终收敛到所选版本，冲突
  与待同步数量归零。
- 删除路径必须由一端写云 tombstone，另一端同步后移除本机测试歌单。不得
  清理任何非测试歌单或浏览器数据。
- 未登录、本机队列、最近播放、播放进度、ChKSz API 密钥/API 地址和设备
  设置继续保持本机边界，不纳入任何跨设备断言或云 payload。
- 若真实验收暴露产品缺陷，停止验收、保留两端数据与证据，建立修复并重新
  跑本里程碑；不能用人工改 Supabase 数据绕过产品行为。

## Acceptance Criteria

- [x] A/B 初始均为同一真实账号、`synced`、待同步 0、冲突 0。
- [x] A 创建含至少 3 首歌的唯一测试歌单并同步；B 收到相同 id/名称/顺序。
- [x] B 改变歌曲顺序并同步；A 刷新/同步后收到相同顺序。
- [x] B 断网再修改时本地立即保留、待同步大于 0；联网后归零且 A 收到。
- [x] B 离线旧版本与 A 在线新版本产生真实冲突；冲突面板出现前，两份数据
  均有证据且没有静默覆盖。
- [x] 明确解决冲突后 A/B 收敛到所选版本，待同步 0、冲突 0、最近成功更新。
- [x] A 或 B 删除测试歌单后另一端同步移除；非测试歌单数量和内容未改变。
- [x] Trellis 记录时间线、设备/浏览器、状态、测试歌单标识、截图或 DOM
  证据和清理结果，但不记录凭据或会话值。
- [x] 自动化桌面/手机账号回归与完整 `npm run verify` 仍通过。
- [ ] 验收证据以独立提交推送，Pages 成功并线上只读复核后，才进入回收站与
  历史版本里程碑。

## Out Of Scope

- 为验收新增生产测试开关、后门 API 或管理员查询。
- 回收站 UI、历史快照、差异预览或自助恢复实现。
- 同步播放队列、播放进度、最近播放或 API 配置。
